威脅分析: `telnet` (不安全的明文協定)

【極度危險警告】絕對不要在現代網路上使用 Telnet

telnet 是一個歷史遺留的協定。它在 1969 年被設計出來，當時的網路環境是基於「信任」的，完全沒有考慮到安全問題。

telnet 協定的所有傳輸（包括使用者名稱、密碼和您輸入的所有指令）都是**未經加密的純文字 (Plain Text)**。

在任何網路上使用 telnet，都等於是**公開廣播**您的登入憑證給任何可能在監聽的人。

telnet 如何運作（以及為何它很危險）

telnet 的基本語法與 ssh 類似，用於連線到遠端主機的指定埠（預設為 23）。

# 語法 (僅為範例，請勿在真實網路使用)
telnet 192.168.1.100

當您執行此指令並連線時，伺服器會要求您輸入使用者名稱和密碼：

Trying 192.168.1.100...
Connected to 192.168.1.100.
Escape character is '^]'.

Login: devuser
Password: [您輸入的密碼]

這與「中間人攻擊」的關係

這正是您之前步驟（`arpspoof` 和 `tcpdump`）所要達成的完美攻擊情境。

如果一個受害者使用 telnet 登入，而攻擊者正在執行「中間人攻擊」並使用 tcpdump 監聽：

  1. 攻擊者執行 `arpspoof`，將自己置於受害者和伺服器之間。
  2. 攻擊者執行 `tcpdump -A -n 'port 23'` 來監聽所有流經的 `telnet` 流量。
  3. 受害者輸入 telnet 192.168.1.100。
  4. 在攻擊者的 tcpdump 螢幕上，將會**即時顯示**如下的**明文**：
...
...
Login: devuser
Password: MySuperSecretPa$$w0rd
...
...
$ ls -la
$ cat /etc/shadow
...
  • 攻擊者**不需要** base64 -d 解碼。
  • 攻擊者**不需要**破解任何東西。
  • 他們**直接**就能看到您的使用者名稱、您的密碼，以及您在登入後輸入的**每一個指令**和伺服器回傳的**每一個結果**。

防禦與結論

  • 永不使用 Telnet: 除非您是在一個完全離線、安全的實驗室環境中操作 30 年前的老舊設備，否則沒有任何理由使用 `telnet`。
  • 停用 Telnet 服務: 作為網路管理員，您的首要任務之一就是確保您所有的伺服器和網路設備（如交換器、路由器）都**關閉**了 telnet 服務。
  • 【解決方案】永遠使用 `ssh`:

    ssh (Secure Shell) 是 `telnet` 的現代、安全替代品。它提供了強大的加密功能，可以完全抵禦我們所描述的竊聽攻擊。即使攻擊者攔截了您的 `ssh` 流量，他們看到的也只是一堆無法解讀的加密亂碼。
